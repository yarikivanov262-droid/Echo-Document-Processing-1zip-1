[ИСПРАВЛЕНО В ЭТОЙ СЕССИИ] Критический баг №1 — Rate-limit блокировал весь API
--------------------------------------------------------------------------------
Где:      artifacts/api-server/src/routes/index.ts
Этап:     STAGE 4 / STAGE 36 (анти-спам)
Симптом:  В логах сервера — сплошной поток HTTP 429 на /api/chats,
          /api/folders, /api/users, /api/users/me и т.д. Через несколько
          секунд обычной работы (список чатов обновляется раз в 3 сек)
          приложение переставало отвечать почти на все запросы.
Причина:  Лимитеры подключались как `router.use(authRateLimit, authRouter)`
          без указания пути. В Express такой вызов означает «применить
          authRateLimit к КАЖДОМУ запросу, проходящему через роутер»,
          а не только к маршрутам внутри authRouter. В результате счётчик
          лимита на логин (10 запросов/15 мин) реально считал ВСЕ запросы
          от пользователя ко всем эндпоинтам сразу (chats, folders, users...),
          и после десятка обычных действий сервер начинал отвечать 429
          на всё подряд.
Первое исправление (недостаточное): лимитеры сначала были перенесены
          внутрь конкретных роутеров через `authRouter.use(authRateLimit)`
          и аналогично для messages.ts/files.ts. При повторном нагрузочном
          тесте (20 быстрых запросов подряд к /api/chats) баг воспроизвёлся
          СНОВА — потому что все роутеры в проекте монтируются в index.ts
          БЕЗ префикса пути (`router.use(authRouter)`, `router.use(messagesRouter)`
          и т.д.), а значит `router.use(limiter)` внутри любого из них
          всё ещё срабатывает на 100% запросов, проходящих через общую
          цепочку middleware, а не только на «свои» маршруты.
Финальное исправление (сделано и проверено нагрузочным тестом): лимитер
          навешен точечно на каждый конкретный маршрут как последний
          аргумент перед обработчиком — например
          `router.post("/login", authRateLimit, async (req, res) => {...})`,
          аналогично для /register, /check-username, /restore в auth.ts,
          для POST /messages в messages.ts и для POST /files/upload в
          files.ts. Проверено вручную через curl: 20 подряд идущих
          запросов к /api/chats, /api/users/me и /api/folders — все
          вернули 200, ни одного 429. Обычная работа приложения (список
          чатов, обновляющийся каждые 3 сек) больше не задевает лимитер логина.

--------------------------------------------------------------------------------
[ИСПРАВЛЕНО В ЭТОЙ СЕССИИ] Критический баг №2 — Загрузка файлов падала с 413
--------------------------------------------------------------------------------
Где:      artifacts/api-server/src/app.ts
Этап:     STAGE 10 (вложения)
Симптом:  PayloadTooLargeError / HTTP 413 при отправке файла через
          /api/files/upload.
Причина:  Файлы передаются как base64 внутри JSON-тела, а
          `express.json()` был подключён без параметра `limit`
          — лимит по умолчанию всего 100kb. Любой файл/фото крупнее
          ~75kb (с учётом base64-раздувания) отклонялся ещё до того,
          как долетал до обработчика.
Исправление (сделано): `express.json({ limit: "25mb" })` и
          `express.urlencoded({ extended: true, limit: "25mb" })`
          в app.ts. Теперь фото и небольшие файлы должны проходить.
Осталось сделать: 25mb — временный практичный лимит. Правильное решение
          на будущее — перейти на multipart/form-data или прямую
          загрузку в объектное хранилище вместо base64-в-JSON, это
          снизит нагрузку на сервер и на базу данных (файлы сейчас,
          похоже, хранятся как base64-строки прямо в БД — см. ниже).

Баг №4 — Тихие отказы (мутации без обработки ошибок)
--------------------------------------------------------------------------------
Действия выполняют запрос к серверу, но если сервер вернул ошибку —
пользователь не получает вообще никакой обратной связи, интерфейс
выглядит так, будто всё прошло успешно:
  - chat-list.tsx: архивирование/разархивирование, mute/unmute, закрепление
    чата в контекстном меню — мутации без onError.
  - create-group.tsx и new-chat.tsx: создание группы/чата — только onSuccess,
    при ошибке кнопка просто «зависает», без сообщения об ошибке.
Как исправить: добавить `onError` в каждый `useMutation`, показывающий
toast с текстом ошибки (`error.message` из ответа API). Это единообразный,
быстрый фикс — можно сделать глобальный дефолтный onError в QueryClient
(`defaultOptions.mutations.onError`) и переопределять только там, где
нужна кастомная реакция.

Баг №5 — Проглатывание ошибок в try/catch без сообщения пользователю
--------------------------------------------------------------------------------
  - lib/ws-client.ts: если пришло сообщение по WebSocket с некорректным
    JSON, ошибка парсинга проглатывается молча — можно потерять событие
    (новое сообщение, статус звонка и т.п.) без всякого лога.
  - routes/backup.ts (backend): ошибки при создании резервной копии
    проглатываются.
  - settings/backup.tsx: сбор сообщений для бэкапа через Promise.all
    в try/catch, при ошибке просто пропускает часть данных — бэкап
    может получиться неполным без предупреждения пользователя.
Как исправить: как минимум логировать ошибку через console.error /
серверный logger, а в backup-флоу — показывать пользователю
предупреждение «часть данных не удалось сохранить» вместо тихого
пропуска.

Баг №6 — Хрупкое переподключение WebSocket
--------------------------------------------------------------------------------
Где:      lib/ws-client.ts, lib/obfuscated-ws.ts
Симптом:  При ошибке сокета обработчик `onerror` просто закрывает
          соединение (`this.ws?.close()`), рассчитывая, что `onclose`
          запустит реконнект. В части браузеров/сетевых условий `onclose`
          может не сработать сразу после `onerror`, из-за чего клиент
          «зависает» без сообщений/звонков до ручного обновления страницы.
          `ObfuscatedWebSocket` (обфусцированный режим для обхода
          блокировок, STAGE 30) вообще не содержит встроенной логики
          переподключения — полагается на потребителя.
Как исправить: добавить явный таймаут в onerror, который форсирует
переподключение, даже если onclose не пришёл; перенести общую логику
экспоненциального backoff-реконнекта в обёртку, используемую и
обычным, и обфусцированным WS-клиентом, чтобы оба режима переподключения
вели себя одинаково надёжно.

Баг №7 — Заглушки с "хардкодом" вместо реальных данных
--------------------------------------------------------------------------------
  - settings/data.tsx: «Экспорт данных» формирует статичный JSON только
    с датой и версией — реальные сообщения/настройки пользователя в
    экспорт не попадают. Функция выглядит рабочей, но бесполезна.
  - settings/data.tsx: «Очистить кэш» чистит только localStorage,
    сохраняя токен, но не трогает IndexedDB (там теперь хранятся приватные
    ключи Signal Protocol и медиа-кэш) — то есть часть данных всё равно
    останется.
  - settings/security.tsx: «Сжечь аккаунт» передаёт в API захардкоженную
    строку `"confirm-burn"` вместо реального подтверждения (например,
    повторного ввода seed-фразы) — по факту любой может нажать кнопку
    без криптографического подтверждения личности.
  - settings/security.tsx: если сервер не вернул список активных сессий,
    интерфейс подставляет фейковую запись «current-device» — пользователь
    может решить, что это единственная сессия, хотя это просто заглушка.
Как исправить:
  1. Экспорт данных — реально выгружать историю чатов пользователя
     (через уже существующие API messages/chats) в JSON.
  2. Очистка кэша — дополнительно вызывать очистку IndexedDB-хранилищ
     (кроме таблицы с приватными ключами, которую трогать нельзя).
  3. Burn account — потребовать повторный ввод seed-фразы на клиенте,
     считать её хэш и отправлять реальный seedHash, а не константу.
  4. Список сессий — при пустом ответе показывать явное сообщение
     «нет данных о сессиях», а не выдуманную запись.

Баг №9 — Слишком агрессивный лимит на анонимные сообщения
--------------------------------------------------------------------------------
Где:      middlewares/rate-limit.ts (anonInboxRateLimit)
Этап:     STAGE 34 (анонимные ссылки)
Симптом:  Лимит — 3 запроса в минуту на IP. Любой пользователь за NAT
          (например, весь офис или мобильный оператор с CGNAT) исчерпывает
          лимит после 3 сообщений от РАЗНЫХ людей и получает отказ.
Как исправить: увеличить окно/порог (например, 3 запроса в минуту на
связку IP+slug получателя, а не просто на голый IP), либо перейти на
более мягкий лимит (10-15/мин) с дополнительной защитой через
капчу/задержку при подозрительной активности.

Баг №10 — Слабая типизация в критичных местах (`as never`)
--------------------------------------------------------------------------------
Где:      chat-list.tsx (опции refetchInterval для React Query)
Симптом:  Приведение к `as never` отключает проверку типов именно там,
          где легко случайно передать не тот параметр в хук запроса —
          ошибка не будет поймана компилятором.
Как исправить: завести собственный типизированный wrapper над
хуками из api-client-react с корректной сигнатурой опций (см. заметку
в памяти агента про Orval query options typing), чтобы не приходилось
использовать `as never` в компонентах.
