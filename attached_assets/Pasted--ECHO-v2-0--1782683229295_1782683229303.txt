📋 ECHO — Итоговый технический план (v2.0)

Слоган: «Говори. Никто не узнает.»
Тип: Полностью анонимный мессенджер с E2EE, Tor, PWA, клон всех функций Telegram (кроме подарков и криптокошелька).
Цель: Единый веб-клиент для всех платформ, работающий через браузер, с установкой на главный экран.

---

1. Общая архитектура

Компонент Технология Назначение
Клиент PWA (React 18 + TypeScript, Vite, Redux Toolkit) Интерфейс, локальное шифрование, Tor-прокси, офлайн-режим
Сервер Go 1.21+ (Gin/Echo, Gorilla WebSocket) REST API, WebSocket-маршрутизация, бизнес-логика
База данных PostgreSQL 16 + pgcrypto Хранение пользователей, сессий, зашифрованных сообщений, метаданных
Файловое хранилище MinIO (S3-совместимое) Зашифрованные медиафайлы, аватарки, стикеры
Сеть Встроенный Tor (WASM) + обфускация Анонимизация IP, обход блокировок, скрытый сервис (.onion)
Криптография Signal Protocol + Web Crypto API + BIP39/BIP32 Сквозное шифрование, PFS, seed-фразы, локальное шифрование
Кэш/Сессии Redis (опционально) Ускорение доступа к сессиям, снижение нагрузки на БД
Планировщик Cron (Go) Очистка старых сессий, удаление аккаунтов, очистка файлов

2. Клиентская часть (PWA)

2.1. Стек и инструменты

· Язык: TypeScript
· Фреймворк: React 18
· Состояние: Redux Toolkit + RTK Query (кэширование API)
· UI: Кастомная компонентная система на styled-components / Material-UI (адаптив)
· Сборка: Vite + PWA-плагин (vite-plugin-pwa)
· Service Worker: для офлайн-кэширования статики и фоновых обновлений
· Tor: @jamescoyle/wasm-tor или самосборный Tor в WASM
· Криптография: libsignal-protocol, bip39, @scure/bip32, argon2-browser
· Хранилище: localForage + шифрование записей AES-256-GCM через Web Crypto API
· Медиа: MediaRecorder API, WebRTC, Lottie для анимированных стикеров

2.2. Структура маршрутов (все страницы)

2.3. Основные UI-компоненты

· Header: логотип, индикатор безопасности (цветной замок), название экрана, поиск, аватар пользователя.
· Sidebar (десктоп): список чатов, кнопка создания, фильтры.
· BottomNavigation (мобильные): «Чаты», «Контакты», «Скрытые» (замок), «Настройки».
· ChatList: карточки чатов с аватаркой, именем, последним сообщением, временем, непрочитанными, закреплением.
· MessageInput: поле ввода, кнопки голосового, вложений, стикеров, таймер самоуничтожения.
· MessageBubble: отображение всех типов сообщений (текст, медиа, голос, видео, файл, контакт, геолокация) с реакциями, ответами, статусами.
· Modal, Toast, LoadingIndicator – общие элементы.

2.4. Локальное хранилище (IndexedDB)

· Схема: объектные хранилища для сообщений, контактов, настроек, логов, черновиков, кэша стикеров и медиа.
· Шифрование: каждая запись шифруется AES-256-GCM с ключом, полученным из пароля (PBKDF2/Argon2). Ключ хранится в сессионной памяти (или в зашифрованном виде в localStorage при опции «запомнить меня»).
· Разделение профилей: для второго профиля используется отдельное хранилище с другим ключом.

2.5. Tor и обход блокировок

· Встроенный прокси: WASM-версия Tor создаёт локальный SOCKS5-прокси; все запросы (HTTP, WebSocket) идут через него.
· Обфускация: при невозможности прямого соединения с .onion используется WebTunnel или поддельный TLS (SNI) для маскировки под обычный HTTPS.
· Ручная настройка: пользователь может указать свой SOCKS5-прокси.

2.6. Криптографические операции на клиенте

· Seed-фраза: генерация BIP39 (12 слов), отображение при регистрации.
· Деривация ключей: BIP32 из seed → identity_key (Ed25519), signed_prekey, одноразовые преключи.
· Signal Protocol: установка сессий, шифрование/дешифрование сообщений, управление преключами.
· Шифрование файлов: случайный ключ AES-256-GCM, ключ шифруется публичным ключом получателя и передаётся с сообщением.
· Хэширование seed: Argon2id для отправки seed_hash на сервер (сервер хранит только хэш).

3. Серверная часть (Go)

3.1. Стек

· Web-фреймворк: Gin или Echo
· WebSocket: Gorilla WebSocket
· База данных: PostgreSQL 16 (драйвер pgx)
· Кэш: Redis (опционально)
· Файловое хранилище: MinIO (или AWS S3)
· Планировщик: robfig/cron

3.2. Модель данных PostgreSQL (основные таблицы)

users

· id BIGSERIAL PRIMARY KEY (автоинкремент, новые пользователи получают id ≥ 100000)
· username VARCHAR(32) UNIQUE NOT NULL
· public_identity_key BYTEA NOT NULL
· public_signed_prekey BYTEA
· public_one_time_prekeys JSONB
· seed_hash BYTEA NOT NULL (Argon2id)
· is_admin BOOLEAN DEFAULT FALSE (для id < 100000)
· is_deleted BOOLEAN DEFAULT FALSE
· created_at TIMESTAMPTZ DEFAULT NOW()
· last_active TIMESTAMPTZ DEFAULT NOW()
· settings JSONB

sessions

· device_id UUID PRIMARY KEY
· user_id BIGINT REFERENCES users(id) ON DELETE CASCADE
· public_device_key BYTEA NOT NULL
· last_used TIMESTAMPTZ DEFAULT NOW()
· created_at TIMESTAMPTZ DEFAULT NOW()
· ip_hash BYTEA (опционально)
· user_agent TEXT

messages

· id BIGSERIAL PRIMARY KEY
· sender_id BIGINT NOT NULL REFERENCES users(id)
· receiver_id BIGINT NOT NULL REFERENCES users(id)
· chat_type SMALLINT (1-private, 2-group, 3-channel, 4-secret)
· chat_id BIGINT (для групп/каналов ссылается на chats.id)
· encrypted_content BYTEA NOT NULL (зашифровано клиентом)
· timestamp TIMESTAMPTZ DEFAULT NOW()
· delivered_at TIMESTAMPTZ
· read_at TIMESTAMPTZ
· delete_after_read BOOLEAN DEFAULT FALSE
· delete_after_seconds INT
· is_voice BOOLEAN DEFAULT FALSE
· media_file_id UUID

chats (для групп/каналов)

· id BIGSERIAL PRIMARY KEY
· type SMALLINT (2-group, 3-channel)
· title VARCHAR(255)
· creator_id BIGINT REFERENCES users(id)
· created_at TIMESTAMPTZ DEFAULT NOW()
· avatar_file_id UUID
· settings JSONB

chat_members

· chat_id BIGINT REFERENCES chats(id) ON DELETE CASCADE
· user_id BIGINT REFERENCES users(id) ON DELETE CASCADE
· role VARCHAR(20) DEFAULT 'member'
· joined_at TIMESTAMPTZ DEFAULT NOW()
· permissions JSONB

pending_deletions

· user_id BIGINT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE
· scheduled_at TIMESTAMPTZ DEFAULT NOW()

backups (опционально)

· id UUID PRIMARY KEY
· user_id BIGINT REFERENCES users(id)
· file_id UUID
· created_at TIMESTAMPTZ DEFAULT NOW()

3.3. API-эндпоинты (REST)

Метод Путь Описание
POST /api/register Регистрация (username, public_key, prekeys, seed_hash) → user_id
POST /api/check_username Проверка доступности username
POST /api/login Создание сессии (device_key, подпись) → device_id, session_token
POST /api/restore Восстановление по seed_hash → новая сессия
POST /api/delete_account Сожжение аккаунта (подпись)
POST /api/upload_file Загрузка зашифрованного файла → file_id
GET /api/download_file/:file_id Скачивание файла
GET /api/users/:username Поиск пользователя (публичный ключ)
POST /api/update_prekeys Обновление одноразовых преключей
POST /api/backup/upload Загрузка бэкапа
GET /api/backup/latest Получение последнего бэкапа

3.4. WebSocket (/ws)

· Авторизация: через session_token в query или первом сообщении.
· Типы сообщений:
  · send — отправка зашифрованного сообщения
  · delivery_ack — подтверждение доставки
  · read_ack — подтверждение прочтения
  · delete_message — команда удаления
  · typing — уведомление о наборе текста
  · status_update — изменение статуса (онлайн/офлайн)
· Маршрутизация: сервер пересылает сообщения активным получателям; офлайн-сообщения сохраняются в БД и доставляются при подключении.

3.5. Планировщик (cron-задачи)

· Ежедневно 03:00: удаление сессий старше 30 дней.
· Еженедельно 04:00: окончательное удаление аккаунтов с is_deleted=TRUE и last_active < NOW() - 2 years.
· Ежедневно 05:00: удаление «сирот» (файлов, не привязанных к сообщениям) старше 7 дней.

4. Безопасность и приватность (ключевые аспекты)

Уровень Меры
Транспорт Все соединения через Tor (.onion), WSS/TLS 1.3, обфускация трафика
Аутентификация Seed-фраза + пароль; сервер хранит только хэш seed; сессии привязаны к устройству через подпись
Сообщения Сквозное шифрование (Signal Protocol, PFS); сервер получает только шифротекст
Файлы Шифруются на клиенте (AES-256-GCM), ключи передаются по E2EE
Локальное хранилище IndexedDB шифруется с помощью Web Crypto (ключ от пароля)
Истории, стикеры Также шифруются и хранятся на сервере в зашифрованном виде
Журнал активности Ведётся локально в зашифрованной БД, не передаётся на сервер
Самоуничтожение Сообщения удаляются по таймеру как локально, так и на сервере
Скрытые чаты Дополнительный PIN-код, двойное шифрование, не синхронизируются с сервером в открытом виде
Сожжение аккаунта Мгновенное удаление всех данных на сервере и клиенте
Анонимность Нет номера телефона, email; регистрация по username; Tor скрывает IP

5. Полный перечень функций (клон Telegram)

Все функции Telegram, кроме подарков и криптокошелька, переносятся:

5.1. Общение и чаты

· Личные, групповые (до 200 000 участников), каналы, секретные чаты.
· Текстовые, голосовые, видеосообщения (кружки), фото, видео, файлы (до 2 ГБ), контакты, геолокации.
· Редактирование и удаление для всех, ответы, цитирование, реакции (кастомные и анимированные).
· Отложенная отправка, самоуничтожение по таймеру, режим «Не беспокоить».

5.2. Форматирование и медиа

· Жирный, курсив, зачёркнутый, моноширинный, спойлер, ссылки, списки.
· Расширенное форматирование для ботов (коллажи, карусели, таблицы, сворачиваемые блоки).
· Встроенный редактор фото/видео (рисование, текст, стикеры), AI-редактор (перевод, трансформация), GIF-редактор.

5.3. Стикеры и эмодзи

· Обычные, анимированные, видео-стикеры; создание стикерпаков.
· Кастомные эмодзи (Premium).
· Поиск стикеров и эмодзи (в т.ч. через ИИ).

5.4. Группы и каналы

· Администрирование (назначение админов, передача владения, медленный режим).
· Закреплённые сообщения, опросы, викторины, планы (Checklists).
· Темы (ветки обсуждений) в группах.
· Статистика для каналов и групп.
· Защита (боты-Guardians), модерация (теги участников).

5.5. Боты и мини-приложения

· Полноценные боты с кнопками, клавиатурами, расширенным форматированием.
· Платформа Mini Apps (запуск веб-приложений внутри).

5.6. Звонки и истории

· Голосовые и видеозвонки (индивидуальные, групповые) с реакциями.
· Истории (Stories) с фото/видео, реакциями, просмотрами.
· Прямые эфиры в историях.

5.7. Аккаунт и устройства

· Несколько аккаунтов (профилей) – быстрое переключение.
· Поддержка всех платформ через PWA.
· Облачное хранение файлов (зашифрованное).
· Настройки приватности (видимость статуса, фото и т.д.).
· Двухфакторная аутентификация (Passkeys) – опционально.

6. Инфраструктура и рекомендации по железу

6.1. Сервер (для 5 000 активных пользователей)

· CPU: 4–8 ядер (например, Intel Xeon / AMD EPYC)
· RAM: 16–32 ГБ (рекомендуется 32 ГБ для кэширования БД)
· Диск: SSD NVMe (не менее 100 ГБ, с учётом роста)
· Сеть: 1 Гбит/с
· ОС: Ubuntu 22.04 LTS
· База данных: PostgreSQL 16 с настроенными параметрами (shared_buffers, effective_cache_size, work_mem), обязательно использовать PgBouncer для пула соединений.
· Tor: Настроить скрытый сервис для .onion-доступа к серверу.

6.2. Дополнительно

· Мониторинг: Prometheus + Grafana (бесплатно).
· Логирование: ELK-стек или Loki (бесплатные аналоги).
· Бэкапы: Автоматическое резервное копирование БД и файлов (например, pg_dump + MinIO).

7. Этапы разработки (для работы с ИИ)

1. Настройка инфраструктуры: Установка Ubuntu, PostgreSQL, MinIO, Tor, настройка .onion.
2. Разработка серверного ядра: Go-сервер с базовыми эндпоинтами (регистрация, логин, WebSocket).
3. Создание клиентского каркаса: React-приложение с роутингом, Redux, подключение к серверу через WebSocket.
4. Интеграция криптографии: BIP39, генерация ключей, Signal Protocol, шифрование IndexedDB.
5. Интеграция Tor: Встройка WASM-версии Tor, настройка проксирования всех запросов.
6. Реализация UI всех страниц и функций: чаты, сообщения, контакты, настройки.
7. Добавление дополнительных фич: голосовые/видеозвонки (WebRTC), истории, стикеры, группы/каналы, боты.
8. Тестирование и аудит безопасности: проверка уязвимостей, тесты на проникновение.
9. Развёртывание и бета-тестирование: запуск для ограниченного круга, сбор обратной связи.
10. Релиз и маркетинг.

9. Заключение

Echo — это полностью анонимный, защищённый от блокировок мессенджер с единой веб-кодовой базой, копирующий все функции Telegram (кроме подарков и криптокошелька). Благодаря использованию Tor, Signal Protocol, seed-фраз и локального шифрования, он обеспечивает максимальную приватность при привычном интерфейсе. Разработка возможна с помощью бесплатных ИИ-инструментов и открытого ПО, а для 5 000 пользователей достаточно умеренного серверного железа (4–8 ядер, 16–32 ГБ RAM, SSD).
